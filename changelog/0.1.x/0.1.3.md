# 0.1.3 — 2026-06-28

## Fixed

- **`gnomad_get_coverage` / `gnomad_list_gene_variants`** — an inverted region (start > stop, e.g. `1-55064852-55039447`) is now rejected up front as a `ValidationError` (`reason: invalid_region`, `retryable: false`) in the service's region parse, before any upstream request. gnomAD answers an inverted range with HTTP 500, which the retry layer read as a transient fault — so the malformed input previously burned the full retry budget and surfaced as a misleading "unavailable" error. A single-position region (start == stop) stays valid. [#1](https://github.com/cyanheads/gnomad-genetics-mcp-server/issues/1)
- **`gnomad_get_variant`** — `GNOMAD_MAX_VARIANT_BATCH` now drives the batch cap, which was hard-coded to 25. The configured value (default 25) flows into the advertised input schema (`variants.maxItems` in `tools/list`), parse-time `.max()` validation, and the `variants` field and tool description prose. [#4](https://github.com/cyanheads/gnomad-genetics-mcp-server/issues/4)
- **`gnomad_variant_triage`** — the callability step now confirms the *exact* variant position instead of gene-level aggregate coverage. A chrom-pos-ref-alt input derives the single-position region (`gnomad_get_coverage(region: "1-55051215-55051215", …)`); an rsID routes through the `variant_id` resolved in step 1; gene-level coverage is offered only as a weaker fallback that does not confirm the position. Dataset values are quoted in every tool-call example so the chains are copyable verbatim. [#6](https://github.com/cyanheads/gnomad-genetics-mcp-server/issues/6)
