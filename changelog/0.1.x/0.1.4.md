# 0.1.4 — 2026-06-30

## Fixed

- **`gnomad_get_coverage`** — region coverage is now bounded to the requested `start..stop` span. gnomAD pads a `region(...)` query to a fixed ~151 bp window centered on the request, so coverage bins previously carried ±~75 bp of neighboring bases into the summary; a single-position region now summarizes exactly the requested base (one bin) rather than its neighborhood. Gene and transcript coverage — the intended whole-feature set — is unchanged. [#8](https://github.com/cyanheads/gnomad-genetics-mcp-server/issues/8)
- **`gnomad_search_clinvar`** — an Ensembl gene ID (`ENSG…`) now returns a targeted notice naming the cause and short-circuits before the NCBI query, instead of running a guaranteed-empty search and surfacing the generic "verify the gene symbol" notice. ClinVar's `[gene]` index resolves HGNC symbols only, so an ENSG ID — valid on every other gnomAD tool — always yielded zero records; the notice points to `ensembl_lookup_gene` to resolve the symbol. The `gene` input description carries the same HGNC-only constraint as a tool-scoped override that leaves the shared field the other tools reuse untouched. [#9](https://github.com/cyanheads/gnomad-genetics-mcp-server/issues/9)

## Changed

- **`@cyanheads/mcp-ts-core`** — `^0.10.9 → ^0.10.10`.

## Security

- **js-yaml** — the transitive `js-yaml` dependency (pulled by the `depcheck` dev dependency) resolved to the patched `3.15.0`, clearing [GHSA-h67p-54hq-rp68](https://github.com/advisories/GHSA-h67p-54hq-rp68) (CVE-2026-53550) — a quadratic-complexity denial-of-service in YAML merge-key handling via repeated aliases. Fell out of the `@cyanheads/mcp-ts-core` `^0.10.10` lockfile re-resolve; `bun audit` now reports no vulnerabilities (was one moderate advisory).
